La seconda parte del progetto è consistita nella realizzazione di un servizio CORBA che fornisse accesso in lettura ad un database.

Il database viene per semplicità realizzato con un file testo in cui sono memorizzate corrispondenze del tipo chiave-valore.

Il servizio viene implementato sfruttando partizionamento e replicazione dei dati su server diversi.

Il  servizio deve poter essere registrato ed utilizzato attraverso l'interfaccia analizzata al punto precedente.

Dal lato client non c'è nessuna coscienza del fatto che la risposta venga da un unico server o che essa sia il risultato del coordinamento di un cluster di servitori.

Il client vede un'interfaccia ma non ha conoscenza di come essa venga in realtà implementata (trasparenza e separazione di interfaccia da implementazione).

Si realizza l'astrazione di risorsa unica.

I vari server che forniscono il servizio costituiscono un cluster. L'idea è quella di permettere che il numero di server costituenti il cluster non sia prefissato a priori; un server potrà quindi aggiungersi in maniera dinamica al cluster fornendo dati più aggiornati e/o ulteriori dati.

L'unica operazione disponibile per un cliente è quella di lettura di una entry della tabella, dato in ingresso un valore di chiave.

Ogni server possiede localmente una tabella che contiene un sottoinsieme delle entry complessive gestite dal sistema.

Si vuole anche realizzare un meccanismo di caching per cui se il client effettua una richiesta di lettura ad un dato ad un server che non lo possiede quest'ultimo lo cerca negli altri server e se tale dato esiste in almeno un agente lo memorizza nella propria tabella (che mantiene in memoria). Se le richieste sono molte e sempre diverse il meccanismo può portare alla replicazione della maggior parte delle entry su tutti i server. Più verosimilmente la maggior parte delle richieste riguarderà un'insieme ristretto delle entry; in questo caso il meccanismo di caching consente risposte più veloce (si evita di dover interrogare altri server).

La replicazione e il partizionamento sono da noi introdotti per raggiungere obiettivi di:
load balancing
fault tolerance

E' necessario inoltre realizzare un meccanismo per cui i dati mantenuti in server diversi siano sempre consistenti fra loro (cioè non esistano server che a uno stesso valore di chiave associano valori differenti).

Il protocollo realizzato è indipendente dal grado di replicazione (che è, in generale, variabile dinamicamente).

Se un server si guasta semplicemente i suoi dati non saranno più disponibili. Può capitare che tale server abbia dati comunque mantenuti in altri server o può capitare che tale server abbia dei dati non posseduti da nessun altro server. Nel primo caso non si ha perdita di informazione, nel secondo tali dati sono non disponibili finché il server non viene riattivato.
